# Release supply-chain artifacts for v1.0.0

## Status

**Proposed — recorded, not yet enforced.** This ADR captures a v1.0.0 release-pipeline architecture decision surfaced by external code review (item H6.a). It stays `proposed` until an independent multi-agent review (`ras consider`) concurs. Acceptance settles the signing/SBOM/SLSA artifact set for v1.0.0; future releases follow the same pattern unless re-opened.

## Context

`.github/workflows/release.yml` is named "Release Validation" and runs on `push: tags: ['v*']`. The current workflow validates the release candidate by running `go test`, `go test -race`, `govulncheck`, and `gosec`. It does **not** produce any signed, attested, or accompanying release artifact:

- No annotated/signed tag verification step (`git verify-tag` is run by consumers manually per `docs/release-verification.md`).
- No SBOM (CycloneDX or SPDX).
- No SLSA provenance.
- No cosign / Sigstore signing of source archives or built artifacts.
- No GitHub Release object created automatically.

`docs/release-checklist.md` §7 defers SBOM and signing to "if a future release adds compiled assets" — a defensible policy for the source-only `v0.x` releases but increasingly out of step with OpenSSF Best Practices and Scorecard expectations as the project approaches v1.0.0.

`docs/release-verification.md` documents the consumer-side verification flow: download the signed tag, verify with `git verify-tag` against the maintainer's SSH allowed-signers file. This is the *currently shipped* trust anchor for cpace releases. It is real — every release tag is in fact signed — but it is enforced by consumer discipline rather than by anything CI emits.

The project is a source-only Go module: there are no compiled binaries to sign. The artifacts at release time are:

- The Git tag (annotated, signed via SSH).
- The Git tree at that tag (the source code).
- Go module proxy entries (`proxy.golang.org`), which derive deterministically from the tag.

Three classes of supply-chain artifact are candidates for v1.0.0:

**Class 1 — Tag and signature verification in CI**
- `git verify-tag` step in `release.yml` against a checked-in allowed-signers file.
- Validates that the workflow only proceeds for tags signed by the documented maintainer key.
- Catches any case where a malicious actor with `Actions: write` somehow triggered the workflow without a valid tag signature.

**Class 2 — Software Bill of Materials**
- SBOM generated by `anchore/syft` in CycloneDX 1.5 (or SPDX 2.3) format.
- Lists every Go module in the dependency graph at the released tag with version and license.
- Attached to the GitHub Release as an asset.
- Validates an OpenSSF Scorecard SBOM check; surfaces dependency provenance to downstream SBOM consumers (regulated environments, government procurement).

**Class 3 — SLSA Build Provenance**
- SLSA v1.0 provenance generated by `slsa-framework/slsa-github-generator`.
- Attests *which workflow run, on which commit, at which time* produced the release artifacts.
- For a source-only release, the "artifact" being attested is the Git tree at the tag plus the SBOM.
- Validates SLSA Build Level 3 with the recommended workflow.

The project's threat model (`docs/threat-model.md`) does not currently treat downstream consumers as the trust audience for cpace's CI infrastructure — the threat model is about the cryptographic protocol surface. Supply-chain attestations are about the *distribution* surface. Adding them is a scope expansion of what cpace warrants about how it ships.

A non-negotiable constraint from project policy: tag-signature verification in CI must not change the existing signed-tag-is-the-artifact contract for `v0.x` consumers. SBOM and SLSA are additive; they do not replace the signed Git tag.

## Decision

Adopt **Class 1 + Class 2 for v1.0.0**. Defer Class 3 (SLSA provenance) to a v1.x.x minor release once the SLSA generator's source-only-module support has been validated against this project.

Concretely:

- **Class 1: Add `git verify-tag` to `release.yml`.** The workflow checks out the tag, then verifies the tag's SSH signature against a checked-in allowed-signers file (already in the repo at the path documented in `docs/release-verification.md`). The workflow fails before any other validation if `git verify-tag` fails. This locks the existing signed-tag policy into CI rather than leaving it consumer-side-only.
- **Class 2: Generate and attach an SBOM.** Add `anchore/syft` step generating a CycloneDX 1.5 JSON SBOM. Upload as workflow artifact. Attach to the GitHub Release object (which the workflow now also creates) on tag pushes that pass all gates.
- **Class 3: Defer.** SLSA Build Level 3 for source-only Go modules has known rough edges with `slsa-framework/slsa-github-generator`. Adding it now risks shipping a fragile attestation that downstream consumers cannot easily verify. Re-open this ADR when SLSA's source-only-module flow is stable, or when a downstream consumer explicitly requests it.

The workflow also gains a "create GitHub Release" step that publishes the release notes from `CHANGELOG.md` for the tagged version, attaches the SBOM, and links to `docs/release-verification.md`.

## Acceptance criteria

The implementation must satisfy these before this ADR moves `proposed → accepted` *and* before v1.0.0 is tagged:

- **`git verify-tag` runs first** in `release.yml` on any `v*` tag push. Workflow fails if the tag is not annotated or not signed by an allowed-signer in the checked-in file. Lightweight tags trigger the workflow but immediately fail this step.
- **SBOM generation** via `anchore/syft@<sha-pin>` produces a CycloneDX 1.5 JSON file. The file is checksummed (SHA-256) and the checksum is recorded in the release notes.
- **GitHub Release object** is created automatically for the tag, with:
  - Title: the tag name.
  - Body: the relevant `CHANGELOG.md` section for that version.
  - Assets: the SBOM file plus its SHA-256 checksum.
- **`docs/release-verification.md` is updated** to describe SBOM verification (compare downloaded SBOM SHA-256 against the value in release notes).
- **`docs/security-gates.md` and `docs/release-checklist.md` are updated** to list SBOM as a release deliverable for v1.x.
- **No SLSA provenance** is generated. The ADR explicitly defers it; the next-step entry in `docs/release-checklist.md` says so.
- **Action SHA pinning** for `anchore/syft` and any other new actions, matching the existing project convention.
- **Workflow permissions** at job level: `contents: write` (for Release creation), `id-token: write` (for any signing later), `attestations: write` (for SBOM attestation). All other defaults remain `contents: read`.

## Considered options

- **A — Class 1 + Class 2, defer SLSA (recommended).** Verifies tag signatures in CI and provides SBOM to downstream consumers. Does not commit to SLSA tooling that may need rework.

- **B — Class 1 + Class 2 + Class 3 (full OpenSSF posture).** All three artifact classes from the start. Larger CI surface, more dependency on the SLSA generator's quirks. If a SLSA-generator regression breaks releases, recovery requires temporary disabling.

- **C — Class 1 only.** Adds tag-signature verification but keeps the source-only-no-extra-artifacts policy. Simplest, no SBOM for downstream consumers.

- **D — Status quo.** Defer all of Class 1, 2, and 3 to a later release. Continue to rely on consumer-side `git verify-tag` discipline.

- **E — Cosign artifact signing.** Add `cosign sign` over a source-archive tarball plus the SBOM. Establishes a Sigstore-anchored trust path for the assets. Has the same dependency-on-tooling-maturity question as SLSA; not all downstream consumers currently verify Sigstore signatures for Go modules.

## Consequences

- **Option A (recommended):**
  - One CI run produces a signed-tag verification artifact (pass/fail) plus an SBOM.
  - GitHub Release object becomes the canonical "what shipped" pointer.
  - Two new action dependencies (`syft` plus a release-action of choice), both pinnable by SHA.
  - Downstream consumers gain SBOM visibility without needing to verify Sigstore signatures.
  - SLSA can be added in a v1.x.x minor release as additive metadata.
  - Adds OpenSSF Scorecard points for SBOM, packaging, and tagging checks.

- **Option B:**
  - All three artifact classes from v1.0.0. Strongest supply-chain posture.
  - Highest dependency on third-party tooling. If `slsa-framework/slsa-github-generator` has a regression for source-only modules, the release pipeline breaks.
  - Mitigation: gate SLSA generation behind a workflow input flag so it can be disabled per-release.

- **Option C:**
  - Verifies tag signature in CI; that alone closes the highest-leverage hardening from the review (item H6.a minimum).
  - No SBOM for downstream consumers; Scorecard SBOM check stays failing.
  - Smaller CI surface to maintain.

- **Option D:**
  - Zero CI change. Signed-tag verification remains consumer-side.
  - Loses ground against OpenSSF Best Practices expectations for v1.0.0.

- **Option E:**
  - Sigstore signatures over source archive + SBOM.
  - Trust anchored in Sigstore root rather than the maintainer's SSH key.
  - Requires consumers to install `cosign` and trust Sigstore's root; not all downstream Go consumers do.
  - Useful complement to options A/B/C, not a replacement.

## Implementation outline (Option A)

1. Add a `verify-tag` job to `.github/workflows/release.yml` that:
   - Runs first, before `test` / `race` / `govulncheck` / `gosec`.
   - Checks out with `fetch-depth: 0` so the tag object is available.
   - Loads the allowed-signers file from the checked-in path (`.github/allowed_signers` or `docs/release-verification.md`-referenced path).
   - Runs `git config gpg.ssh.allowedSignersFile <path>` and `git verify-tag "$GITHUB_REF_NAME"`.
   - Fails the workflow if `verify-tag` exits non-zero.
2. Add an `sbom` job (or append to `release-validation`) that:
   - Uses `anchore/syft` pinned by SHA.
   - Generates `cpace-<version>.cdx.json` (CycloneDX 1.5 JSON).
   - Uploads as a workflow artifact.
3. Add a `release` job that:
   - Runs after all gates pass.
   - Uses `gh release create` or a pinned third-party release-action.
   - Body content: the relevant `CHANGELOG.md` section for the version, extracted by a small script.
   - Assets: the SBOM and its SHA-256 checksum.
4. Update `docs/release-verification.md` to add SBOM verification (download the SBOM, compute SHA-256, compare to release notes value).
5. Update `docs/release-checklist.md` §7 to remove the "if a future release adds compiled assets" qualifier and replace with the new policy.
6. Update `docs/security-gates.md` "Release Validation" row to list SBOM among gates.
7. Add a `CHANGELOG.md` Unreleased entry recording the new release artifacts.
8. **Dry-run** the workflow on a non-`v*` tag (e.g., `test-release-...`) to verify the pipeline works before tagging v1.0.0.
